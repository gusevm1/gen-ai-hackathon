# Plan 24-01: Prompt Injection & Scoring Rules

**Phase:** 24 — Prompt Injection & Scoring Rules
**Goal:** Claude receives pre-fetched nearby places as structured prompt data and scores proximity exclusively from that evidence, with all tool-calling infrastructure removed.
**Requirements:** PROMPT-01, PROMPT-02, PROMPT-03, SCORE-01, SCORE-02

---

## Context

**What exists:**

- `backend/app/services/claude.py` contains `ClaudeScorer.score_listing()`, which runs an agentic loop offering Claude a `search_nearby_places` tool at scoring time. This includes `PLACES_TOOL`, `_AMENITY_KEYWORDS`, `_has_location_criteria()`, and `_execute_tool()`.
- `backend/app/services/proximity.py` already imports `_AMENITY_KEYWORDS` from `claude.py` (line 22) to identify amenity keywords in dynamic fields. This import will break if `_AMENITY_KEYWORDS` is removed from `claude.py` without first relocating it.
- `backend/app/routers/scoring.py` fetches `nearby_data` (step 1b/1c, line 98) before calling `claude_scorer.score_listing()` (step 5, line 142) — but never passes `nearby_data` into the scorer. It only attaches it to `score_data` post-scoring for Supabase persistence.
- `backend/app/prompts/scoring.py` has `build_user_prompt(listing, prefs)` with a single call site in `claude.py` line 151, and `build_system_prompt()` with a `NEARBY PLACES SEARCH:` block (lines 89–93) that instructs Claude to use the now-removed tool.

**What changes:**

1. `_AMENITY_KEYWORDS` moves from `claude.py` to `proximity.py` (fixes import dep before deletion).
2. All tool machinery deleted from `claude.py`; `score_listing()` simplified to a single non-iterative API call.
3. System prompt `NEARBY PLACES SEARCH:` block replaced with `PROXIMITY EVALUATION RULES:` block.
4. `build_user_prompt()` gains a `nearby_places` param; injects "## Nearby Places Data (Verified)" section when data is present.
5. `score_listing()` gains a `nearby_places` param and passes it through to `build_user_prompt()`.
6. Router threads `nearby_data` into `score_listing()`.

**Why this order matters:** Task 1 must precede Task 2 because deleting `_AMENITY_KEYWORDS` from `claude.py` without first defining it in `proximity.py` breaks the running import chain. Tasks 2 and 3 can be done together (same files, same deploy requirement per research risk #7 — system prompt and tool removal must land together). Task 3 (prompt injection + routing) is independent of the tool removal but logically follows it.

```python
    async def score_listing(
        self,
        listing: FlatfoxListing,
        preferences: UserPreferences,
        image_urls: list[str] | None = None,
    ) -> ScoreResponse:
        """Score a listing against user preferences using Claude.

        Makes a single structured API call — no tool use, no agentic loop.
        Proximity evaluation relies exclusively on data injected into the prompt
        by build_user_prompt() (added in Phase 24).

        Args:
            listing: The Flatfox listing to evaluate.
            preferences: The user's search preferences with weights.
            image_urls: Optional list of listing image URLs for visual analysis.

        Returns:
            Validated ScoreResponse with category breakdown and summary.
        """
        client = self.get_client()

        # Build content blocks: images first (if any), then text prompt
        content: list[dict] = []
        content.extend(build_image_content_blocks(image_urls or []))
        content.append(
            {"type": "text", "text": build_user_prompt(listing, preferences)}
        )

        response = await client.messages.parse(
            model=CLAUDE_MODEL,
            max_tokens=4096,
            system=build_system_prompt(preferences.language),
            messages=[{"role": "user", "content": content}],
            output_format=ScoreResponse,
        )

        return response.parsed_output
```

### Task 3: Inject nearby places into prompt and thread through call chain

**File(s):**
- `backend/app/prompts/scoring.py`
- `backend/app/services/claude.py`
- `backend/app/routers/scoring.py`

**What:** Add `nearby_places` parameter to `build_user_prompt()` and `score_listing()`. When `nearby_places` is truthy, inject a "## Nearby Places Data (Verified)" section into the user prompt between the listing data block and the closing evaluation instruction. Thread `nearby_data` from the router into the scorer.

**How:**

**In `backend/app/prompts/scoring.py`:**

1. Update `build_user_prompt()` signature from:
   ```python

   ```
   to:
   ```python
   def build_user_prompt(
       listing: FlatfoxListing,
       prefs: UserPreferences,
       nearby_places: dict[str, list[dict]] | None = None,
   ) -> str:
   ```

2. Update the docstring to add:
   ```
       nearby_places: Optional dict mapping amenity query string to list of place
           result dicts. When provided, appended as a verified data section before
           the closing evaluation instruction.
   ```

3. The function currently returns a single f-string ending at line 344. Change the return to build the string conditionally. Replace the final `return f"""..."""` structure with:

```python
    # Build the base prompt (preferences + listing data + closing instruction)
    base = f"""## User Preferences
...
(keep the existing f-string body exactly as-is, ending with the closing instruction line)
...
Evaluate this listing against the user's preferences. Score each of the 5 categories \
(location, price, size, features, condition), evaluate the custom criteria and desired features \
as a checklist, and provide an overall score with summary bullets highlighting key matches \
and compromises."""

    # Append verified nearby places section when pre-fetched data is available (PROMPT-01, PROMPT-02)
    if nearby_places:
        lines = ["\n\n---\n\n## Nearby Places Data (Verified)\n"]
        lines.append(
            "The following results were fetched from Google Places before this evaluation.\n"
            "Evaluate all proximity-based criteria EXCLUSIVELY from this data.\n"
        )
        for query, places in nearby_places.items():
            lines.append(f"\n### {query}")
            if places:
                for i, place in enumerate(places, start=1):
                    name = place.get("name", "Unknown")
                    dist = place.get("distance_km")
                    rating = place.get("rating")
                    review_count = place.get("review_count")
                    address = place.get("address", "")
                    dist_str = f"{dist:.2f} km" if dist is not None else "distance unknown"
                    rating_str = (
                        f" | Rating: {rating} ({review_count} reviews)"
                        if rating is not None
                        else ""
                    )
                    lines.append(f"{i}. {name} — {dist_str}{rating_str}")
                    if address:
                        lines.append(f"   {address}")
            else:
                lines.append("No results found.")
        lines.append(
            "\n\nIf an amenity is NOT listed above, treat it as \"not found nearby\" "
            "— do not guess or infer from listing description."
        )
        return base + "\n".join(lines)

    return base
```

   Concretely: extract the existing f-string into a variable named `base`, then conditionally append and return. The existing f-string body (lines 303–344) is unchanged — only the `return` statement changes.
