# Phase 01.1: Build Custom Homegate Scraper Actor on Apify - Context

**Gathered:** 2026-03-05
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the third-party Apify actor (`ecomscrape/homegate-property-search-scraper`) with a custom-built Apify actor for scraping Homegate property listings. The custom actor must produce output compatible with the existing `normalizeHomegate` pipeline and `PropertyListingSchema` validation.

</domain>

<decisions>
## Implementation Decisions

### Research & foundation approach
- Fork & adapt: Research existing Homegate scrapers on GitHub, Apify Store, and open-source projects
- Find the best existing scraper and adapt it into a custom Apify actor rather than building from scratch
- Prioritize battle-tested field mappings and proven DOM/API interaction patterns

### Claude's Discretion
- **Field extraction depth**: Determine which fields are reliably extractable based on research findings. Current schema supports ~15 fields (price, rooms, address, livingSpace, floor, yearBuilt, description, images, features, propertyType, agency, coordinates) — extract what's practical
- **Page depth strategy**: Determine whether to scrape search result pages only or also follow to detail pages, based on what existing scrapers do and field availability at each level
- **Scraper selection criteria**: Weigh recency/maintenance, data completeness, and Apify/Crawlee ecosystem fit when choosing which existing scraper to base the actor on
- **Scraping technology**: Choose between Cheerio, Playwright, or Puppeteer based on Homegate's rendering requirements (determined during research)
- **Anti-bot handling**: Determine appropriate rate limiting, proxy usage, and retry strategies based on research findings

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `HomegateAdapter` (src/scrapers/homegate/adapter.ts): Calls Apify actor and retrieves dataset items — will need actor ID updated to custom actor
- `normalizeHomegate` (src/scrapers/homegate/normalize.ts): Maps ~15 fields with broad fallback chains — actor output should align with these field names where possible
- `PropertyListingSchema` (src/schema/listing.ts): Zod schema with required fields (url, title, listingType, price, rooms, address) — actor must produce data that passes validation
- `parseSwissPrice/Rooms/Area` (src/parsers/swiss-numbers.ts): Swiss number format parsers — actor can output raw Swiss-format strings and let normalize.ts handle parsing

### Established Patterns
- ScraperAdapter interface: `{ name: string; scrape(options): Promise<RawRecord[]> }` — adapter pattern is fixed
- RawRecord = `Record<string, unknown>` — generic shape, normalize.ts handles field extraction
- Apify integration via `apify-client` package — actor.call(input) then dataset.listItems()

### Integration Points
- `HomegateAdapter.scrape()`: Change actor ID from `ecomscrape/homegate-property-search-scraper` to custom actor
- Actor input format: Currently `{ urls: [...], maxItems?: number }` — custom actor can define its own input schema
- `manual-run.ts`: Entry point that wires adapter → normalize → validate → write

</code_context>
