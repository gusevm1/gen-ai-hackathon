## Summary

Homegate.ch is Switzerland's largest real estate marketplace (100K+ properties, 200M+ monthly page views). The site uses Vue.js with server-side rendering (SSR) and embeds structured JSON data directly into HTML `<script>` tags via `window.__INITIAL_STATE__` (search pages) and `window.__PINIA_INITIAL_STATE__` (detail pages). This means property data can be extracted by parsing the HTML and extracting JSON from script tags -- no browser rendering required in principle, though proxy rotation is recommended to avoid anti-bot measures.

The best existing reference implementation is the **Scrapfly homegate-scraper** (Python, open source on GitHub), which demonstrates the exact JSON paths, pagination logic, and dual-page extraction pattern (search pages vs. detail pages). This should be adapted into a TypeScript Apify actor using **Crawlee's CheerioCrawler** (fast, lightweight, no browser overhead) with Apify Proxy as a fallback if plain HTTP requests get blocked.

**Primary recommendation:** Build a CheerioCrawler-based Apify actor that extracts `window.__INITIAL_STATE__` JSON from search result pages, with pagination via the `?ep=` URL parameter. Start with search-page-only extraction (covers most fields). Fall back to PlaywrightCrawler only if CheerioCrawler gets blocked by anti-bot measures.

### Pattern 1: JSON-from-Script-Tag Extraction (PRIMARY)
**What:** Extract structured JSON data embedded in `<script>` tags rather than parsing DOM elements.
**When to use:** When the target site uses SSR with embedded state (Vue/Nuxt, React/Next, etc.).
**Why:** More reliable than CSS selectors (class names change on redesign), provides complete data in one extraction.

```typescript
// Source: Scrapfly homegate-scraper (adapted to TypeScript/Cheerio)
// https://github.com/scrapfly/scrapfly-scrapers/tree/main/homegate-scraper

import type { CheerioAPI } from 'cheerio';

function parseInitialState($: CheerioAPI): Record<string, unknown> | null {
  const scriptContent = $('script')
    .filter((_, el) => {
      const text = $(el).html() || '';
      return text.includes('window.__INITIAL_STATE__');
    })
    .first()
    .html();

  if (!scriptContent) return null;

  // Format: window.__INITIAL_STATE__={...json...}
  const jsonStr = scriptContent.split('=').slice(1).join('=').trim();
  return JSON.parse(jsonStr);
}
```

### Pattern 2: Search Page Pagination
**What:** Iterate through search result pages using the `?ep=` query parameter.
**When to use:** Scraping all listings from a search query.

```typescript
// Source: Scrapfly homegate-scraper pattern, adapted for Crawlee
// JSON path: data.resultList.search.fullSearch.result

const data = parseInitialState($);
const result = data?.resultList?.search?.fullSearch?.result as any;
const listings = result?.listings ?? [];
const pageCount = result?.pageCount ?? 1;

// Enqueue remaining pages on first page
if (request.userData.page === undefined || request.userData.page === 1) {
  const requests = [];
  for (let page = 2; page <= pageCount; page++) {
    const pageUrl = new URL(request.url);
    pageUrl.searchParams.set('ep', String(page));
    requests.push({ url: pageUrl.toString(), userData: { page } });
  }
  await crawler.addRequests(requests);
}
```

### Anti-Patterns to Avoid
- **CSS selector extraction:** Do NOT use `.HgListingCard_price_JoPAs` or similar class selectors. These are generated class names that change on every build/deploy. Use the `__INITIAL_STATE__` JSON extraction instead.
- **Single page scraping without queue:** Do NOT fetch pages in a loop with `fetch()`. Use Crawlee's request queue for automatic retries, rate limiting, and proxy rotation.
- **Hardcoding page count:** Do NOT loop `for page 1..100`. Extract `pageCount` from the JSON response and paginate dynamically.
- **Ignoring Actor lifecycle:** Do NOT skip `Actor.init()` and `Actor.exit()`. Without these, the actor won't integrate with Apify storage or proxy.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| HTTP retries & backoff | Custom retry loops | CheerioCrawler built-in retries (`maxRequestRetries: 3`) | Handles timeouts, 429s, network errors with exponential backoff |
| Proxy rotation | Manual proxy list cycling | `Actor.createProxyConfiguration()` | Apify manages proxy pool, session stickiness, rotation on errors |
| Request deduplication | URL set tracking | Crawlee's RequestQueue | Persistent queue survives crashes, handles dedup automatically |
| Rate limiting | `setTimeout` delays | CheerioCrawler `maxConcurrency` + `minConcurrency` | Framework-level concurrency control with adaptive throttling |
| Data persistence | `fs.writeFileSync` to JSON | `Actor.pushData()` to Dataset | Apify Dataset supports pagination, export to JSON/CSV/Excel, API access |
| Pagination management | Manual page counter | Crawlee `addRequests()` with `userData` | Queued requests survive crashes, avoid duplicate pages |

**Key insight:** Crawlee/Apify SDK handles all the infrastructure concerns (retries, proxies, queuing, persistence). The actor code should focus purely on extraction logic.

## Common Pitfalls

### Pitfall 1: CheerioCrawler Blocked by Anti-Bot
**What goes wrong:** Homegate may return empty pages, CAPTCHAs, or 403 errors when detecting non-browser HTTP clients.
**Why it happens:** Cloudflare or similar protection detects missing browser fingerprint in plain HTTP requests.
**How to avoid:** Start with CheerioCrawler + Apify residential proxies (`countryCode: 'CH'`). If blocked, upgrade to PlaywrightCrawler with the same logic. Design extraction code to be crawler-agnostic.
**Warning signs:** Empty `__INITIAL_STATE__`, HTTP 403/429, HTML containing "checking your browser" text.

### Pitfall 2: JSON Parse Failures on __INITIAL_STATE__
**What goes wrong:** `JSON.parse()` throws on malformed or truncated script content.
**Why it happens:** The `split('=', 1)` approach can fail if the JSON itself contains `=` signs. Script tag may contain comments or multiple assignments.
**How to avoid:** Use `split('=').slice(1).join('=')` to handle `=` inside JSON. Add try/catch with logging. Validate the parsed object has expected structure before processing.
**Warning signs:** Increasing parse error rate in logs.

### Pitfall 3: Missing Fields Between Search and Detail Pages
**What goes wrong:** Search page listings have fewer fields than detail pages (e.g., no description, no features, no yearBuilt).
**Why it happens:** Search pages show compact listing cards; full data is only on detail pages at `/rent/LISTING_ID` or `/buy/LISTING_ID`.
**How to avoid:** For Phase 01.1, start with search-page-only extraction. Map which fields are available from search JSON vs. which require detail pages. The normalize.ts already handles optional fields gracefully.
**Warning signs:** High rate of `undefined` for optional fields like description, features, yearBuilt.

### Pitfall 4: Actor ID Mismatch After Deployment
**What goes wrong:** The `HomegateAdapter` still calls the old `ecomscrape/homegate-property-search-scraper` actor ID.
**Why it happens:** Forgetting to update the actor ID in the adapter after deploying the custom actor.
**How to avoid:** The custom actor deploys under the user's namespace (e.g., `username/homegate-scraper`). Update `HomegateAdapter` to use the new actor ID. Make the actor ID configurable via environment variable.
**Warning signs:** Still getting billed for the third-party actor.

### Pitfall 5: Exceeding Apify Free-Tier Credits
**What goes wrong:** CheerioCrawler with residential proxies burns through credits faster than expected.
**Why it happens:** Residential proxy usage costs significantly more than datacenter. Each page request with proxy counts against credit balance.
**How to avoid:** Start with datacenter proxies (cheaper). Use residential only if datacenter gets blocked. Set `maxRequestsPerCrawl` to limit run size during development. Monitor credit usage in Apify Console.
**Warning signs:** Credit balance dropping faster than expected in Apify dashboard.

### Pitfall 6: Stale CSS Class Selectors
**What goes wrong:** Scrapers that use `.HgListingCard_price_JoPAs` break after Homegate deploys.
**Why it happens:** CSS module hashes change on every build. This is why the GitHub homegate scraper repos use outdated class names.
**How to avoid:** NEVER use CSS class selectors for data extraction. Use `__INITIAL_STATE__` JSON extraction exclusively.
**Warning signs:** If you see class names with random suffixes like `_JoPAs` or `_K9dLF` in your code, you're doing it wrong.

### Detail Page JSON Structure (via __PINIA_INITIAL_STATE__)
```typescript
// Source: Scrapfly homegate-scraper parse_property_page function
// https://github.com/scrapfly/scrapfly-scrapers/blob/main/homegate-scraper/homegate.py
//
// JSON path from window.__PINIA_INITIAL_STATE__:
// data.listing.listing

function parsePiniaState($: CheerioAPI): Record<string, unknown> | null {
  const scriptContent = $('script')
    .filter((_, el) => {
      const text = $(el).html() || '';
      return text.includes('window.__PINIA_INITIAL_STATE__');
    })
    .first()
    .html();

  if (!scriptContent) return null;

  const match = scriptContent.match(
    /window\.__PINIA_INITIAL_STATE__\s*=\s*(\{.+\})\s*$/s
  );
  if (!match) return null;

  const data = JSON.parse(match[1]);
  return data?.listing?.listing ?? null;
}
```

### Complete Actor main.ts Pattern
```typescript
// Source: Crawlee docs + Apify platform deployment guide
// https://crawlee.dev/js/docs/deployment/apify-platform

import { Actor } from 'apify';
import { CheerioCrawler, log } from 'crawlee';
import type { CheerioAPI } from 'cheerio';

interface ActorInput {
  listingType: 'rent' | 'buy' | 'both';
  maxItems?: number;
}

await Actor.init();

const input = await Actor.getInput<ActorInput>();
const listingType = input?.listingType ?? 'both';
const maxItems = input?.maxItems;

const proxyConfiguration = await Actor.createProxyConfiguration({
  // Start with datacenter; switch to RESIDENTIAL if blocked
});

const crawler = new CheerioCrawler({
  proxyConfiguration,
  maxConcurrency: 5,
  maxRequestRetries: 3,
  requestHandlerTimeoutSecs: 60,

  async requestHandler({ request, $, crawler: c }) {
    const state = parseInitialState($);
    if (!state) {
      log.warning(`No __INITIAL_STATE__ found on ${request.url}`);
      return;
    }

    const result = (state as any)?.resultList?.search?.fullSearch?.result;
    const listings = result?.listings ?? [];
    const pageCount = result?.pageCount ?? 1;

    // Push extracted listings to dataset
    for (const listing of listings) {
      await Actor.pushData(transformListing(listing, request.url));
    }

    // Enqueue remaining pages (only on first page)
    if (!request.userData.page || request.userData.page === 1) {
      const limit = maxItems ? Math.min(pageCount, Math.ceil(maxItems / 20)) : pageCount;
      for (let page = 2; page <= limit; page++) {
        const url = new URL(request.url);
        url.searchParams.set('ep', String(page));
        await c.addRequests([{
          url: url.toString(),
          userData: { page },
        }]);
      }
    }
  },
});

// Build start URLs
const startUrls: string[] = [];
if (listingType === 'rent' || listingType === 'both') {
  startUrls.push('https://www.homegate.ch/rent/real-estate/switzerland/matching-list');
}
if (listingType === 'buy' || listingType === 'both') {
  startUrls.push('https://www.homegate.ch/buy/real-estate/switzerland/matching-list');
}

await crawler.run(startUrls);
await Actor.exit();
```

## Discretion Recommendations

### Field Extraction Depth
**Recommendation: Extract all available fields from search page JSON.** The `window.__INITIAL_STATE__` search results appear to contain: id, listingType, characteristics (rooms, livingSpace, floor, yearBuilt), address (street, zip, city, coordinates), prices (rent/buy amount, currency), title, and images. Description and features may only be available on detail pages.

**Confidence: MEDIUM** -- JSON structure inferred from Scrapfly scraper patterns and multiple sources. Exact field availability needs live verification during development.

### Page Depth Strategy
**Recommendation: Search pages only for Phase 01.1.** Search pages provide enough fields to satisfy the required PropertyListingSchema fields (url, title, listingType, price, rooms, address). Optional fields like description and features can be added later via detail page scraping. This keeps the actor simple and fast (20 listings/page, ~50 pages = ~1000 listings in ~50 requests vs ~1050 requests with detail pages).

### Scraper Selection Criteria
**Recommendation: Use the Scrapfly homegate-scraper as the reference implementation.** It is:
- Open source (GitHub, MIT-compatible)
- Recently maintained
- Uses the JSON extraction approach (not fragile CSS selectors)
- Documents the exact JSON paths for both search and detail pages
- Demonstrates pagination handling

The other GitHub repo (bahramkhanlarov) uses outdated CSS class selectors and is Python/BeautifulSoup only. The Apify ecomscrape actor is closed-source and costs $20/month.

### Scraping Technology
**Recommendation: Start with CheerioCrawler (no browser).** Homegate uses Vue.js SSR, which embeds `__INITIAL_STATE__` in the initial HTML response. CheerioCrawler can parse this without JavaScript execution. This is 10-50x faster and uses far less memory/credits than browser-based crawling.

If CheerioCrawler gets blocked (403s, CAPTCHAs, empty states), the fallback plan is:
1. Add Apify residential proxies with `countryCode: 'CH'`
2. If still blocked, switch to PlaywrightCrawler (same extraction logic, just different crawler class)

### Anti-Bot Handling
**Recommendation:**
- `maxConcurrency: 5` (conservative, ~5 concurrent requests)
- `maxRequestRetries: 3` with Crawlee's built-in exponential backoff
- Start with datacenter proxies (cheaper); upgrade to residential if needed
- Set appropriate `User-Agent` headers via Crawlee's session management
- Monitor for 403/429 responses and empty `__INITIAL_STATE__` as detection signals

## Open Questions

1. **Exact `__INITIAL_STATE__` JSON schema on live Homegate pages**
   - What we know: Scrapfly scraper uses `data.resultList.search.fullSearch.result.listings[]` path and `data.listing.listing` for detail pages
   - What's unclear: Exact field names and nesting may have changed since Scrapfly's implementation. The title/description fields may use locale-keyed objects (e.g., `{ de: "...", fr: "..." }`)
   - Recommendation: First task in implementation should be a "discovery run" -- fetch one search page and one detail page, dump the full JSON, and map fields to PropertyListingSchema

2. **Does CheerioCrawler work without proxies?**
   - What we know: Scrapfly uses `render_js: True` and `asp: True` (anti-scraping protection bypass), suggesting some protection exists. But SSR sites typically serve complete HTML including script tags.
   - What's unclear: Whether plain HTTP GET without proxies returns full HTML including `__INITIAL_STATE__`
   - Recommendation: Test locally with a simple fetch first. If blocked, add proxies. If still blocked, switch to PlaywrightCrawler.

3. **How many listings per search page?**
   - What we know: The `pageCount` field exists in JSON. Other scrapers paginate with `?ep=` parameter.
   - What's unclear: Exact number of listings per page (likely 20, common for real estate sites)
   - Recommendation: Determine from first page response during discovery run

4. **Actor deployment namespace**
   - What we know: `apify push` deploys under the authenticated user's namespace
   - What's unclear: The exact actor ID format (e.g., `username/homegate-scraper`)
   - Recommendation: After `apify push`, note the full actor ID and update `HomegateAdapter` accordingly
