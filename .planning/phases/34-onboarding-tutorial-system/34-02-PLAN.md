**1a. Add onboarding message handlers to `background.ts`:**

Add two new message types to the `ExtMessage` union:
```typescript
| { action: 'getOnboardingState' }
| { action: 'updateOnboardingState'; step: number; active: boolean; completed: boolean }
```

Add two new cases to the `handleMessage` switch:

```typescript
case 'getOnboardingState': {
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) return { onboarding: null };
  const { data } = await supabase
    .from('profiles')
    .select('preferences')
    .eq('is_default', true)
    .single();
  return { onboarding: data?.preferences?.onboarding ?? null };
}
case 'updateOnboardingState': {
  const { step, active, completed } = message as {
    action: 'updateOnboardingState';
    step: number;
    active: boolean;
    completed: boolean;
  };
  const { error } = await supabase.rpc('update_onboarding_state', {
    p_step: step,
    p_active: active,
    p_completed: completed,
  });
  return { error: error ? { message: error.message } : null };
}
```

The `getOnboardingState` handler first checks auth -- if no user is logged in, return null (no onboarding). The `updateOnboardingState` handler calls the Supabase RPC function deployed in Plan 01.

**1b. Create `extension/src/lib/onboarding.ts`:**

Content-script-friendly helpers that communicate with the background script via `browser.runtime.sendMessage`:

```typescript
export interface OnboardingState {
  onboarding_step: number;
  onboarding_active: boolean;
  onboarding_completed: boolean;
}

export async function getOnboardingState(): Promise<OnboardingState | null> {
  try {
    const response = await browser.runtime.sendMessage({ action: 'getOnboardingState' });
    return response?.onboarding ?? null;
  } catch {
    return null;
  }
}

export async function updateOnboardingState(
  step: number,
  active: boolean,
  completed: boolean,
): Promise<void> {
  await browser.runtime.sendMessage({
    action: 'updateOnboardingState',
    step,
    active,
    completed,
  });
}
```

Wrap in try/catch because the content script may load before the background script is ready. Return null on error -- the content script should treat null as "no onboarding active."

**2b. Create `OnboardingOverlay.tsx`:**

A full-viewport overlay with SVG mask spotlight cutout:

Props: `targetSelector: string | null`, `targetShadowHost?: string` (for targeting elements inside shadow roots like the FAB), `step: number`, `totalSteps: number`, `title: string`, `instruction: string`, `onNext: () => void`, `onSkip: () => void`, `nextLabel?: string`.

Behavior:
1. On mount/prop change, find the target element:
   - If `targetShadowHost` is provided, find the shadow host element in the host page DOM to get its bounding rect (for FAB targeting).
   - Otherwise use `document.querySelector(targetSelector)` on the host page DOM.
2. Call `getBoundingClientRect()` to get position.
3. Scroll the element into view: `el.scrollIntoView({ behavior: 'smooth', block: 'center' })`.
4. Render full-screen SVG with mask: dark overlay everywhere except a rounded rectangle cutout over the target.
5. Render `OnboardingTooltip` positioned relative to the cutout rect.
6. Add resize/scroll listeners to reposition (debounced).
7. Handle click on the overlay background (outside cutout) -- do nothing (don't close, let user click Skip explicitly per UX rules).

SVG mask approach (from research):
```tsx
<svg className="absolute inset-0 w-full h-full" style={{ pointerEvents: 'none' }}>
  <defs>
    <mask id="spotlight-mask">
      <rect width="100%" height="100%" fill="white" />
      <rect x={rect.left - 8} y={rect.top - 8}
        width={rect.width + 16} height={rect.height + 16}
        rx="8" fill="black" />
    </mask>
  </defs>
  <rect width="100%" height="100%" fill="rgba(0,0,0,0.5)"
    mask="url(#spotlight-mask)" style={{ pointerEvents: 'auto' }} />
</svg>
```

The cutout area must allow pointer events to pass through so the user can interact with the highlighted element (click the FAB, click a badge, etc.). Set `pointerEvents: 'none'` on the SVG and `pointerEvents: 'auto'` only on the dark overlay rect. Then add a transparent div over the cutout area with `pointerEvents: 'none'` to let clicks through.

Onboarding advance function:
```typescript
const advanceOnboarding = async () => {
  if (!onboardingState) return;
  const next = onboardingState.onboarding_step + 1;
  const completed = next > 8;
  const newState = {
    onboarding_step: completed ? 8 : next,
    onboarding_active: !completed,
    onboarding_completed: onboardingState.onboarding_completed || completed,
  };
  setOnboardingState(newState);
  await updateOnboardingState(newState.onboarding_step, newState.onboarding_active, newState.onboarding_completed);
};
```

Skip function:
```typescript
const skipOnboarding = async () => {
  if (!onboardingState) return;
  const newState = { ...onboardingState, onboarding_active: false };
  setOnboardingState(newState);
  await updateOnboardingState(newState.onboarding_step, false, newState.onboarding_completed);
};
```

**2d. Modify `index.tsx` -- mount onboarding overlay as separate shadow root:**

The onboarding overlay needs its own `createShadowRootUi` mounted AFTER the FAB to ensure higher z-index stacking. However, since the overlay state is managed inside App.tsx, the simpler approach is to render the OnboardingOverlay as a portal-like component WITHIN the existing FAB shadow root. Since both the overlay and FAB share the same React tree in App.tsx, this works naturally.

The overlay uses `position: fixed; inset: 0; z-index: 99999` which will cover the entire viewport even from within the shadow root. This works because fixed positioning is relative to the viewport, not the shadow host.

No changes needed to index.tsx if the overlay renders inside App.tsx's existing shadow root. If z-index conflicts occur with badge shadow roots, create a separate overlay shadow root in index.tsx with position: 'overlay' and z-index: 999999.
